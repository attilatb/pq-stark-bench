- Someone already published the exact thing you were going to publish, five years ago and peer-reviewed: Khaburzaniya et al., "Aggregating and thresholdizing hash-based signatures using STARKs" (https://eprint.iacr.org/2021/1048, AsiaCCS 2022) gives prover wall-clock time, prover RAM, and proof size at batch sizes 128 / 256 / 512 / 1024 on named hardware (8-core Intel Core i9, 32 GB RAM), for a post-quantum signature verified inside a STARK proof, with the code open-sourced in https://github.com/facebook/winterfell. Accessed 2026-07-23.

- The headline answer you were hoping to discover is already known and it is boring: prover time and memory grow in a straight line with batch size, so cost per signature stays flat (roughly 19 to 20 milliseconds per signature at every batch size from 128 to 1024 in https://eprint.iacr.org/2021/1048; 14.29 to 14.77 milliseconds per transaction across batches of 1 to 16 in https://arxiv.org/abs/2603.07974). Only the proof size shrinks per item. Do not pitch "we will show how cost amortizes with batch size" as a finding.

- A live, public, continuously-updated benchmark harness for this exact workload already exists and is run by the Ethereum ecosystem: https://github.com/leanEthereum/leanBench sweeps batches of 125, 250, 500 and 1000 post-quantum signatures, records timing plus CPU and memory per hardware profile, and publishes results at https://bench.leanroadmap.org/. The roadmap at https://leanroadmap.org/ also lists an in-progress track comparing six different proving machines on this same task.

- What is genuinely missing is narrower and less glamorous: nobody has published measured prover cost for the three actual government-standardized algorithms (ML-DSA-44, Falcon-512, SLH-DSA-128s) inside a general-purpose proving machine. The published work all uses custom, proof-friendly signature schemes instead; the only ML-DSA attempts are two hobby repos with zero stars and no measurements (https://github.com/ebulgin/qrypta-pu) and a blog post whose own numbers do not add up and whose linked code repository is empty (https://medium.com/@phillyj1026/building-a-zero-knowledge-verifier-for-dilithium-signatures-ntt-gadget-implementation-in-sp1-6c50ab262836). StarkWare built the Falcon verifier but left the benchmark box explicitly unticked (https://github.com/starkware-bitcoin/s2morrow).

- The novelty claim does not survive as written; all four independent checks refuted it, and a reviewer will produce the 2021 paper within one search. Honest repositioning: "We publish the first measured, standards-conformant prover cost (time, peak memory, proof size, named hardware, runnable code) for ML-DSA-44, Falcon-512 and SLH-DSA-128s inside general-purpose proving machines, with a classical signature control on the same rig." Two disclosures are mandatory or the work gets torn apart: cost per signature is expected to be flat in batch size (cite https://eprint.iacr.org/2021/1048), and no mainstream proving machine has built-in acceleration for these algorithms while it does for the classical ones (https://docs.succinct.xyz/docs/sp1/optimizing-programs/precompiles), so any head-to-head overstates the post-quantum penalty unless said out loud.
