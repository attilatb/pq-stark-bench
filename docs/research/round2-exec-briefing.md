# PQ-STARK-BENCH: Pre-Publication Briefing

Round 2 verification is done. Read this before you publish anything.

## WHAT CHANGED

- **The BTQ/StarkWare date in your brief is wrong by a year, and the "first" is contradicted.** The page says "Sep 18, 2023", not 2024, and BTQ's own words are "the first successful STARK-based verification of post-quantum digital signatures **on Starknet**" — a self-asserted vendor claim, scoped to their own chain. It publishes **zero** proving time, step count, or RAM. It also never says a proof was actually generated, only that a Cairo program was written and profiled. ([btq.com](https://www.btq.com/blog/completing-the-first-falcon-signature-verification-in-starkware-initiating-the-transition-to-a-quantum-safe-ethereum), [eprint.iacr.org/2021/1048](https://eprint.iacr.org/2021/1048))
- **CONTRADICTED: RISC Zero does have Keccak acceleration.** Round 1 said it did not. It ships a patched `tiny-keccak` (behind the `unstable` feature, stabilized in v3.0.1), and Keccak-f[1600] is the exact inner loop of SHAKE. So "RISC Zero has no SHAKE path" is false and would be corrected in public. The real asymmetry is that **neither** prover has any lattice or NTT precompile, while both accelerate Ed25519/ECDSA. ([dev.risczero.com/api/zkvm/precompiles](https://dev.risczero.com/api/zkvm/precompiles))
- **CONTRADICTED: SP1 guests are 64-bit now, not 32-bit.** v6.3.1 source sets `DEFAULT_TARGET = "riscv64im-succinct-zkvm-elf"` (RV64IM, soft-float), while RISC Zero is RV32IM. Succinct's own docs page still says riscv32 and is stale. Consequence: an SP1 cycle and a RISC Zero cycle are different units and can never share a chart axis. ([github source](https://raw.githubusercontent.com/succinctlabs/sp1/refs/tags/v6.3.1/crates/build/src/lib.rs), [stale docs page](https://docs.succinct.xyz/docs/sp1/writing-programs/compiling))
- **UNREACHABLE / DEAD: there is no cost-per-proof number to cite from RISC Zero.** Bonsai was shut down. The docs page 404s, and Boundless states "As of December 2025, Bonsai is no longer available." SP1's network is auction-priced in a token with no published rate. Anyone quoting a dollar figure from either is quoting a dead product. ([docs.boundless.network](https://docs.boundless.network/developers/tutorials/bonsai))
- **A near-miss competitor exists that Round 1 did not have: HAPPIER, hash-based PQ signatures aggregated inside RISC Zero, with a batch sweep from 128 to 262,144.** But it is paywalled (Unpaywall: `oa_status: closed`), publishes no memory and no hardware, and its headline times are **extrapolated** by a formula, from runs that only ever aggregate two real signatures (every log reads `Bitfield: 0b00000011`). ([code and raw logs](https://github.com/ArdaSaygan/PQ-Aggregation-via-Recursive-SNARKs), [paywalled chapter](https://link.springer.com/chapter/10.1007/978-3-032-15541-2_1))

## GO / NO-GO

- **RISC Zero locally on the M3 Max: GO.** arm64 macOS is a first-class supported platform and it automatically uses the Mac's Metal graphics cores for proving. Throw away the first timed run, the docs warn GPU kernels are compiled on first use. ([dev.risczero.com/api/generating-proofs/local-proving](https://dev.risczero.com/api/generating-proofs/local-proving))
- **SP1 locally on the M3 Max: GO-WITH-CAVEAT.** It installs and runs natively (there is a `darwin_arm64` binary in v6.3.1), but there is **no** graphics-card acceleration on Apple hardware at all: CUDA is "only supported on Linux x86_64" and the other speedup is Intel-only. So on this one machine RISC Zero runs accelerated and SP1 runs plain. Fallback: run the head-to-head chart on a Linux x86 box, and treat the Mac as a development machine only. Publishing a Mac-only SP1-vs-RISC-Zero time chart is indefensible. ([hardware acceleration docs](https://docs.succinct.xyz/docs/sp1/generating-proofs/hardware-acceleration), [release v6.3.1](https://github.com/succinctlabs/sp1/releases/tag/v6.3.1))
- **PQ signature libraries inside the guest programs: GO-WITH-CAVEAT, and this is the least-verified item here.** Nobody has published a working ML-DSA-44 verifier in either zkVM, so you are writing it. Two known hazards: guests have no floating-point hardware (SP1's own spec says "soft-float"), and Falcon reference code uses floating point on the signing side, so do not blindly port it. Fallback: use integer-only verification code and prove out one signature end-to-end before touching batching. ([SP1 ABI spec](https://raw.githubusercontent.com/succinctlabs/sp1/refs/tags/v6.3.1/zkevm/README.md))
- **Falcon as a second scheme: GO-WITH-CAVEAT.** Verification is integer-only so it is feasible, but the whole published Falcon record is on-chain execution cost, not proving cost, so you get no baseline to check yourself against. OpenZeppelin publishes Cairo step counts and gas only, and its ML-DSA crate is an explicit stub whose code literally returns true for any correctly-sized input. Fallback: ship ML-DSA-44 first, Falcon second, SLH-DSA third. ([OpenZeppelin/cairo-pq-verifiers](https://github.com/OpenZeppelin/cairo-pq-verifiers))
- **Batch size 1000: GO-WITH-CAVEAT on time, not on memory.** Guest address space is roughly 3 GB and 1000 ML-DSA-44 signatures plus keys is under 4 MB, so the data fits. The risk is prover hours and host RAM: the closest prior art needed 7.4 GB of prover memory for 1024 much cheaper hash-based signatures. Fallback: sweep 1, 2, 4 ... and stop at whatever N still finishes overnight, then publish the curve you actually measured. ([RISC Zero memory layout](https://dev.risczero.com/api/zkvm/zkvm-specification), [ePrint 2021/1048 Table 3](https://eprint.iacr.org/2021/1048))
- **Groth16 wrapping (the small on-chain-verifiable proof): NO-GO on RISC Zero on this Mac, GO-WITH-CAVEAT on SP1.** RISC Zero states flatly that its Groth16 prover "only works on x86 architecture, and so Apple Silicon is currently unsupported (even via Docker)." SP1 documents a Docker path needing about 14 GB for Groth16 (and about 60 GB for the other mode, which will not fit comfortably in 64 GB). Fallback: make the headline metric the STARK proof, and do any Groth16 leg on a Linux x86 machine, clearly labelled as different hardware. ([RISC Zero local proving](https://dev.risczero.com/api/generating-proofs/local-proving), [SP1 hardware requirements](https://docs.succinct.xyz/docs/sp1/getting-started/hardware-requirements))

## THE CLAIM WE CAN ACTUALLY MAKE

**Full version, for the README and any paper:**

> Post-quantum signature verification inside a proof system is not new: hash-based schemes were benchmarked inside a STARK in 2021, and Falcon was implemented in Cairo by BTQ and StarkWare in 2023. What has never been published is a reproducible, multi-prover measurement of prover wall-clock time and peak memory for the three NIST-standardized signature schemes inside a general-purpose zkVM, on named hardware, with the precompile advantage that classical schemes enjoy disclosed in every chart. PQ-STARK-BENCH publishes that, with open code and raw result files.

**Site hero version:**

> What does it actually cost to prove a NIST post-quantum signature inside a zkVM? Nobody had measured it. We did, on two provers, with the code and the raw numbers.

**Phrases that must never appear anywhere:**

- "first" or "world's first" or "first ever", in any construction. ([2021/1048 predates everyone](https://eprint.iacr.org/2021/1048))
- "nobody has benchmarked post-quantum signatures in a STARK". They have.
- "we show how prover cost amortizes with batch size". Three independent sources already show it does not: per-signature prover time is flat and only proof bytes shrink. ([ZK-ACE Table 7](https://arxiv.org/abs/2603.07974), [leanBench results](https://github.com/leanEthereum/leanBench))
- "X times slower than classical", stated without the precompile disclosure sitting next to it.
- "FIPS-conformant Falcon". Falcon is FIPS 206 **draft**, not final. Say "Falcon spec conformant".
- Any dollar-per-proof figure sourced from Bonsai. The product is dead.
- Any sentence putting an SP1 cycle count and a RISC Zero cycle count in the same comparison.

## BIGGEST RISK

- **The single thing most likely to sink this is publishing a prover-versus-prover chart measured on the Mac.** On that machine RISC Zero gets graphics-card acceleration and SP1 gets none, and the two run different instruction sets, so any such chart measures which vendor ships a Mac backend, not which prover is faster. One informed reader posts that and the benchmark's credibility is gone. Mitigation: run every cross-prover comparison on one Linux x86 machine where both have comparable backends, put the hardware, prover version, and precompile list on the face of every chart, and publish the Mac numbers separately and only as a single-prover result. ([SP1 acceleration docs](https://docs.succinct.xyz/docs/sp1/generating-proofs/hardware-acceleration), [RISC Zero local proving](https://dev.risczero.com/api/generating-proofs/local-proving))
