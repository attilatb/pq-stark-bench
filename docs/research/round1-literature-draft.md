# Literature and Prior Art

## 1. Scope and method of this survey

This survey covers published prior art for one question: **what does it cost, in prover time, memory, and proof size, to verify a post-quantum signature inside a STARK or zkVM, and how does that cost behave as the batch size grows?**

Three sub-areas were searched: Falcon/FN-DSA in zero knowledge, ML-DSA/Dilithium in zero knowledge, and hash-based signatures (XMSS, Lamport, SPHINCS+/SLH-DSA) in zero knowledge. Sources were gathered by web search plus direct fetches of IACR ePrint, arXiv, GitHub (including the GitHub REST API for repo metadata), vendor documentation, and vendor blogs. Where a PDF was blocked, it is noted inline.

Method caveats that a reader should hold throughout:

- **Access failures were common and are disclosed, not hidden.** `eprint.iacr.org` PDF endpoints returned HTTP 403 (Cloudflare challenge) to both automated fetchers and curl for several papers. Two of the most important papers were eventually read by loading the PDF in a real browser and extracting the text layer in-page; others were not, and are marked **unverified**.
- **Vendor material is labelled as such.** BTQ, LambdaClass, Succinct, and OpenZeppelin material is flagged so a reader can discount it appropriately.
- **Arithmetic derived from a source's own table is labelled as derived**, not attributed to the source as a stated claim.
- All access dates in this document are **accessed 2026-07-23**.

## 2. What already exists

### 2.1 Native (non-circuit) PQ benchmarks

These establish the floor that any in-circuit number is a multiple of.

- **ML-DSA verification 0.14 ms vs ECDSA 0.88 ms at NIST security level 5, on an ARM-based laptop.** Covers ML-DSA, Dilithium, Falcon, Mayo, SLH-DSA, SPHINCS+, and Cross, with an open-source harness. Source: "Assessing the Impact of Post-Quantum Digital Signature Algorithms on Blockchains", [arXiv:2510.09271](https://arxiv.org/abs/2510.09271v1), 2025-10-10, accessed 2026-07-23. Figures taken from the abstract via search result, not from a fetched full text - **unverified**.
- **Hash-based signature scheme parameters, measured natively.** ePrint 2025/055, published version at [cic.iacr.org/p/2/1/13](https://cic.iacr.org/p/2/1/13) (IACR Communications in Cryptology Vol 2 No 1, published 2025-04-08, DOI 10.62056/aey7qjp10), accessed 2026-07-23. Benchmarked on "a MacBook Pro with Apple M3 Pro chip, 18 GB memory" with Criterion, single-threaded. Target Sum Winternitz w=2, delta=1.1, lifetime 2^18, SHA3-256: keygen 16.39 s, sign 54.79 us, verify 26.37 us, signature 2.22 KiB, 258.75 32-byte-word hashes. The same parameter set under Poseidon2: keygen 166.22 s, sign 888.55 us, verify 351.37 us, signature 2.65 KiB, 105 calls to the width-16 permutation plus 57 calls to width-24. The paper states Poseidon2 is natively about 10x slower than SHA-3, and its advantage is purely in-circuit, which this paper does not measure.
- **SLotH hardware accelerator cycles for SLH-DSA-SHAKE-128f**: keygen 204,310 cycles, signing 4,943,111 cycles, verification 434,660 cycles. Source: [github.com/slh-dsa/sloth](https://github.com/slh-dsa/sloth), accessed 2026-07-23; figures from a search-result summary, repo not fetched directly - **unverified**. These are **silicon** RISC-V cycles on a dedicated hash unit, not zkVM cycles. Conflating the two is the single easiest way to produce a wrong number by several orders of magnitude.
- **Polynomial arithmetic cycle counts for the LaBRADOR-Falcon setting**, the only measured timings in ePrint 2024/311: two-splitting-with-lifting multiplication 1,090 cycles and NTT 9,890 cycles for Falcon-512; 1,850 and 20,694 for Falcon-1024. Hardware verbatim: "GCC 13.2.1 in a 64-bit Intel Core i7-7700K Kaby Lake CPU running at 3.60GHz, with Turbo Boost turned off". Read from the PDF of [eprint.iacr.org/2024/311](https://eprint.iacr.org/2024/311) via in-browser extraction, accessed 2026-07-23; the direct PDF URL returns HTTP 403.
- **Lemur native PQ multi-signature aggregation**, [eprint.iacr.org/2026/1161](https://eprint.iacr.org/2026/1161), received 2026-06-03, accessed 2026-07-23 (full text read via in-browser PDF extraction; direct fetch returns 403). On an AMD Ryzen AI 9 HX 370, 12C/24T, 32 GB, rustc 1.96.0: batch verification of 1,024 signers in 15.0 ms (about 14.6 us per signer), aggregate 185.5 KB, signing 4.2 ms with BDS08 traversal. Values for N above 2^10 are linear extrapolations from an N=2^13 run and are labelled as such in the paper. Code at [github.com/lemur-sig/lemur](https://github.com/lemur-sig/lemur), MIT, accessed 2026-07-23. This is a native scheme with **no proof system at all** and is the strongest "you may not need a zkVM" baseline.

### 2.2 PQ signatures inside proof systems

**Falcon in R1CS.** [github.com/zhenfeizhang/falcon-r1cs](https://github.com/zhenfeizhang/falcon-r1cs), archived 2022-05-17, accessed 2026-07-23. Arkworks R1CS circuits for Falcon verification. Falcon-512: NTT conversion 15,360 constraints, verify-with-NTT 81,460 constraints (1,025 instance / 78,386 witness), verify-with-schoolbook 315,956. Falcon-1024: 30,720 / 162,870 / 1,156,150. README baseline: "an ECC scalar multiplication over Jubjub curve (~256 bits) takes a little over 3k constraints", making Falcon "something like 10 times more costly than proving, say, Schnorr over Jubjub curve". The README also states non-native field ops cost roughly 30x native ones. **No prover time, no proof size, no memory figure exists anywhere in this repo.** Its `falcon-aggregate-sig` crate is a literal `fn main() { println!("Hello, world!"); }` stub, so there is no batching. The successor umbrella repo [github.com/zhenfeizhang/falcon.rs](https://github.com/zhenfeizhang/falcon.rs) (default branch `master`, latest commit 2022-07-05, accessed 2026-07-23) additionally contains a `falcon-plonk` crate on Espresso's Jellyfish backend that publishes **no gate counts at all**.

**Falcon in Cairo (STARK-provable execution).** [github.com/OpenZeppelin/cairo-pq-verifiers](https://github.com/OpenZeppelin/cairo-pq-verifiers), created 2026-06-24, accessed 2026-07-23. Five Falcon-512 verifier variants, measured in L2 gas and Cairo steps against a stated cap of 100,000,000 gas / 1,000,000 steps: Poseidon-native 12,045,449 gas / 97,681 steps; hint variant BLAKE2s 12,809,640 / 104,854; direct BLAKE2s 23,005,760 / 205,781; SHAKE-256 50,163,668 / 312,892; SHAKE-256 direct 60,359,788 / 413,819. **This is the single most useful measured dataset for standards-conformance cost:** the FIPS-conformant SHAKE-256 variant costs 3.2x the steps of the Poseidon-native one. ML-DSA-44 and Poseidon-WOTS+ verifiers exist in the repo but are **stubbed and not measured**. Explicitly experimental and unaudited.

**Falcon and SPHINCS+ in Cairo with Stwo.** [github.com/starkware-bitcoin/s2morrow](https://github.com/starkware-bitcoin/s2morrow), created 2025-04-27, accessed 2026-07-23. Roadmap read via the GitHub API: Falcon512 verification is checked DONE, SPHINCS+ 128s SHA2 is checked DONE, but **"Stwo proving benchmarks" is explicitly UNCHECKED**. StarkWare scoped precisely the benchmark this project proposes and has not published it. Public demo figures at [s2morrow.xyz](https://www.s2morrow.xyz/) (accessed 2026-07-23, no date on page): Falcon-512 about 9.5M L2 gas, secp256r1 syscall about 27.4M, secp256k1 syscall about 16.6M, secp256r1 Garaga about 4.6M, secp256k1 Garaga about 4.5M; 1,030 calldata felts compressed to 62. A search snippet quoting "63K steps" for this demo did **not** appear on the fetched page and conflicts with OpenZeppelin's 97,681 - treat as **unverified**.

**Falcon in Miden VM (MASM).** [github.com/0xPolygonMiden/miden-vm/pull/369](https://github.com/0xPolygonMiden/miden-vm/pull/369), merged 2022-10-12, accessed 2026-07-23. Cycle counts by commit: 335,399 initial, 337,161 after loopification, 347,649 with a pointer-based interface, 337,402 after API conformance. Critically, **signature decompression happens outside the VM** and the four degree-512 polynomials are handed in as memory-resident coefficients, so the measurement covers verification arithmetic only. The shipped stdlib procedure is `std::crypto::dsa::rpo_falcon512`, documented at [0xpolygonmiden.github.io/miden-vm/user_docs/stdlib/crypto/dsa.html](https://0xpolygonmiden.github.io/miden-vm/user_docs/stdlib/crypto/dsa.html) - **that docs URL returned HTTP 404 on fetch, accessed 2026-07-23, so the 337,402 figure there is corroborated only by the PR thread**. This is **RPO-Falcon512**, not FN-DSA: RPO256 replaces SHAKE-256 in hash-to-point specifically to be cheaper in-VM, so the cycle count is not comparable to a FIPS-conformant verifier.

**Falcon aggregation without a circuit.** "Aggregating Falcon Signatures with LaBRADOR", [eprint.iacr.org/2024/311](https://eprint.iacr.org/2024/311), received 2024-02-23, revised 2024-08-09, CRYPTO 2024 pp. 71-106, accessed 2026-07-23 (full text read via in-browser PDF extraction after HTTP 403 on direct fetch). Expresses Falcon verification in LaBRADOR's **native lattice dot-product relation** rather than compiling to R1CS. Falcon-512 aggregate sizes, with salts: 93 kB at N=500, 120 kB at N=1000, 165 kB at N=2000, 252 kB at N=4096, 417 kB at N=8192; without salts, 73 / 80 / 85 / 88 / 89 kB. Security 121 bits for Falcon-512, 249 bits for Falcon-1024. Compression under 13% at N=2000 and under 8% at N=8192. **There is no end-to-end prover implementation, no wall-clock prover or verifier time, and no memory figure anywhere in the paper.** The companion repo [github.com/dfaranha/aggregate-falcon](https://github.com/dfaranha/aggregate-falcon) (accessed 2026-07-23, no LICENSE file, 2 commits) is an **analytic size estimator plus a polynomial-arithmetic microbenchmark**, not a prover. Lemur, citing this work, quotes "around 73 KB for 1024 signers" and notes "no concrete implementation is provided".

**ML-DSA in SP1.** Two zero-star hobby repos and one blog post constitute the entire direct-overlap population.
- [github.com/ebulgin/qrypta-pu](https://github.com/ebulgin/qrypta-pu), created and last pushed 2026-05-19, exactly one commit, 0 stars, accessed 2026-07-23. Verifies one ML-DSA-44 signature (libcrux-ml-dsa 0.0.8) inside SP1 v5.2.4 and wraps to Groth16. README states "~40 min, needs ~14 GB RAM" for CPU Groth16 generation; proof 260 bytes, public values 64 bytes. **No hardware is named, no cycle count is published, and the repo's own test file comments say "~10 min" for CPU proving - internally inconsistent with the README's ~40 min.** Its architecture doc's performance numbers are explicitly marked "(caso publicado)", i.e. secondhand.
- [github.com/vjb/quantum-safe-cre](https://github.com/vjb/quantum-safe-cre), accessed 2026-07-23. Second hobby SP1 ML-DSA implementation. README not fetched; **no numbers, unverified**.
- ["Building a Zero-Knowledge Verifier for Dilithium Signatures"](https://medium.com/@phillyj1026/building-a-zero-knowledge-verifier-for-dilithium-signatures-ntt-gadget-implementation-in-sp1-6c50ab262836), published 2025-12-08, accessed 2026-07-23, and its landing page [dilithium-zk-landing.vercel.app](https://dilithium-zk-landing.vercel.app/), accessed 2026-07-23. Claims 5,625,411 SP1 cycles for ML-DSA-65, 22.07 s proof time, 260-byte proof, about $0.045 per proof. **Do not cite these numbers.** Three independent problems: (a) the component breakdown (~580,000 + ~200,000 + ~320,000) sums to about 1.1M against a stated total of 5,625,411, and 580,000/5,625,411 is 10.3%, not the claimed 50%; (b) the published crate `sp1-ntt-gadget` v0.1.0 contains **no signature verifier at all** - grepping its source for "signature" returns zero matches - and its "SP1 precompile" is a stub whose zkVM and non-zkVM branches call the identical plain-Rust function; (c) the GitHub repo [kota1026/pq-wallet-sp1-ntt-gadget-](https://github.com/kota1026/pq-wallet-sp1-ntt-gadget-) is empty (size 0, 0 stars, created and pushed at the same timestamp, accessed 2026-07-23) and the URL printed in the article body 404s. The gadget's own advertised soundness is 60 bits from 4 challenges, far below the usual 100-128 bit bar.

**Hash-based signature aggregation in STARKs - the strongest and oldest prior art.** "Aggregating and thresholdizing hash-based signatures using STARKs", [eprint.iacr.org/2021/1048](https://eprint.iacr.org/2021/1048), received 2021-08-16, revised 2022-03-14, AsiaCCS 2022 (DOI 10.1145/3488932.3524128), accessed 2026-07-23 (full 15-page PDF read in-browser after the direct URL returned 403). Hardware verbatim: "an 8-core Intel Core i9 processor @ 2.4 GHz with 32 GB of RAM." Table 3, aggregate signatures at 96-bit security: n=128 gives 2.5 s / 0.9 GB / 68 KB; n=256 gives 5.1 s / 1.8 GB / 71 KB; n=512 gives 10.5 s / 3.7 GB / 77 KB; n=1024 gives 19.7 s / 7.4 GB / 83 KB. At 123-bit security: 3.2 s / 1.2 GB / 129 KB through 25.7 s / 9.5 GB / 165 KB. Verification 3 ms to 10 ms. Circuit shape: 22 registers, 1024 steps per signature, max constraint degree 7 for the aggregate AIR. **Derived from their table (my arithmetic, not the paper's claim): amortized prover cost per signature is 19.5, 19.9, 20.5, 19.2 ms at n = 128/256/512/1024 - essentially flat.** The AIR ships in [github.com/facebook/winterfell](https://github.com/facebook/winterfell) (README accessed 2026-07-23), which independently publishes 64 through 1024 signatures at 1.2 s / 0.5 GB / 110 KB up to 20.5 s / 7.6 GB / 152 KB on an i9-9980HK. These two tables do not exactly match (different STARK parameter regimes) and should be cited separately, not merged.

**Hash-based aggregation in a general zkVM.** "HAPPIER: Hash-Based, Aggregatable, Practical Post-quantum Signatures Implemented Efficiently with Risc0", LightSec 2025, [link.springer.com/chapter/10.1007/978-3-032-15541-2_1](https://link.springer.com/chapter/10.1007/978-3-032-15541-2_1), accessed 2026-07-23. Aggregate signatures 2-3 MB, up to 2^16 signatures on a standard laptop; the Groth16 wrapper is disabled to preserve post-quantum security, leaving a pure STARK. Lemur independently cites it as "2.89 MB for 2^16 signatures". **The Springer page redirects to an auth wall and the full text was NOT read; whether it tabulates prover seconds against batch size is unverified and is the largest open risk to any novelty claim in this document.**

**Hash-based batch verification in Winterfell, independent replication.** "Scalable Batch Verification for Post-Quantum Hash-Based Signatures Using STARKs", SBSeg 2025, [sol.sbc.org.br PDF](https://sol.sbc.org.br/index.php/sbseg_estendido/article/download/36772/36558/), DOI 10.5753/sbseg_estendido.2025.11845, accessed 2026-07-23. Proof 50-75 KiB, under 3.1 s for N=64, beats naive per-signature checking for all N >= 8. **Numbers from a search-result summary; the PDF was not fetched - unverified.**

**Purpose-built SNARK-friendly PQ signatures**, which exist precisely because verifying the standardized schemes in-circuit is expensive. CAPSS, [eprint.iacr.org/2025/061](https://eprint.iacr.org/2025/061), accessed 2026-07-23: 24K-35K R1CS constraints with 9.5-15.5 KB signatures at 128-bit security using Anemoi, stated as 5-8x fewer R1CS constraints than Loquat. Loquat, [eprint.iacr.org/2024/868](https://eprint.iacr.org/2024/868), accessed 2026-07-23 (abstract only, no constraint counts extracted). Compact Aggregate Signature from Module-Lattices, [eprint.iacr.org/2023/471](https://eprint.iacr.org/2023/471), accessed 2026-07-23: aggregating 2^20 signatures gives "only several hundred kilobytes", abstract-level only, **PDF returned 403, unverified**.

**A cautionary counter-example.** ZK-ACE, [arXiv:2603.07974](https://arxiv.org/abs/2603.07974) v3 dated 2026-05-28, accessed 2026-07-23 (full PDF read locally). This paper deliberately **removes** PQ signature verification from the circuit and proves an identity-commitment statement instead. Its own measurements are real: Circle STARK via Stwo gives 341 active rows / 361 AIR constraint expressions, 14.5 ms prove, 1.1 ms verify, about 107 KB proof; Groth16/BN254 gives 2,155 R1CS constraints, 37.3 ms prove, 128-byte proof; hardware "single-threaded, Apple M3 Pro", Criterion, 100 samples. Its Table 7 batch sweep is directly relevant: prove/tx at N=1,2,4,8,16 is 14.29, 14.45, 14.46, 15.76, 14.77 ms while proof/tx falls 122 KB to 13 KB. **But its widely quoted "roughly 500-2,300x smaller than in-circuit post-quantum signature verification" is measured-vs-estimated, not measured-vs-measured**: the paper's own Table 10 caption states the ML-DSA and FN-DSA figures are "structural lower bounds based on the algebraic operations required for verification", and only the in-circuit ECDSA row (~1.5M, cited to 0xPARC circom-ecdsa) comes from a real implementation. The frequently repeated "on the order of millions of R1CS constraints" for ML-DSA-44 is this uncited estimate.

### 2.3 PQ chains and state proofs

- **Ethereum's post-quantum consensus effort routes around the NIST lattice schemes entirely.** [pq.ethereum.org](https://pq.ethereum.org/), dated 2026-06-27, accessed 2026-07-23, names hash-based signatures (leanXMSS) with SNARK aggregation via a minimal zkVM (leanVM) for the consensus layer; Dilithium appears once, as a candidate still under evaluation at the execution layer. [ethereum.org/roadmap/future-proofing/quantum-resistance](https://ethereum.org/roadmap/future-proofing/quantum-resistance/), article updated 2026-04-09, accessed 2026-07-23, quotes a hash-based signature at about 3,000 bytes against 96 bytes for BLS, and leanVM "compressing the data by 250x".
- **leanVM / leanMultisig.** [github.com/leanEthereum/leanMultisig](https://github.com/leanEthereum/leanMultisig), accessed 2026-07-23: on an M4 Max 48 GB, CPU only, 1,426 XMSS/second with 327 KiB proofs at WHIR rate 1/2, and 1,027 XMSS/second with 220 KiB proofs at rate 1/4; recursive aggregation over batches of 700 XMSS gives a single batch at 0.22 s / 268 KiB (rate 1/2) and four batches at 0.64 s / 105 KiB (rate 1/4). Security about 124 bits. Stack: WHIR multilinear PCS, SuperSpartan, Logup with an OpenVM-style bus, KoalaBear field.
- **leanBench, an existing running harness for exactly this workload.** [github.com/leanEthereum/leanBench](https://github.com/leanEthereum/leanBench), accessed 2026-07-23. 13 aggregate variants: flat aggregation over 125-1000 signatures plus tree aggregation at several fan-ins, recording sign/verify/aggregate timings plus CPU and memory, grouped by hardware fingerprint, deterministic seed 0xC0FFEE, contributors submit result JSON. **The published dashboard at [bench.leanroadmap.org](https://bench.leanroadmap.org/) rendered only chart scaffolding when fetched on 2026-07-23; no timing values were extractable, so the per-machine results remain unverified from this session.**
- **Roadmap and targets.** [leanroadmap.org](https://leanroadmap.org/), accessed 2026-07-23, tracks pq-devnet-0 (Oct 2025) through pq-devnet-4 (Mar 2026, active, recursive aggregation via leanVM), and lists a research track "Post-Quantum Signature Aggregation with zkVMs" at roughly 50% complete evaluating Binius M3, SP1, KRU, STU, Jolt, and OpenVM. **No results table from that track was findable.** Several leanSig cells on that page rendered as "~0 KiB / ~0 hours / ~0 us" through the fetcher; those are extraction artifacts and are **unverified - do not cite them**. The 2026 plan at [hackmd.io/@tcoratger/ryS1ElrWbx](https://hackmd.io/@tcoratger/ryS1ElrWbx), accessed 2026-07-23, states targets of 1,000 XMSS/s, 2-to-1 recursive aggregation in about 200 ms, and propagation of a 256 KB SNARK in under one second, on KoalaBear with a 128-bit provable security target.
- **Devnet sizing was set by the benchmark.** [pq-devnet-2 spec](https://github.com/leanEthereum/pm/blob/main/breakout-rooms/leanConsensus/pq-interop/pq-devnet-2.md), target 2026-01-28, accessed 2026-07-23: 400 validators, "assuming current benchmark of almost 400 XMSS/s", 4-second slots, five independent clients.
- **Operational proving data.** [LambdaClass, "ethlambda: how we got a 3x speedup in signature aggregation"](https://blog.lambdaclass.com/ethlambda-how-we-got-a-3x-speedup-in-signature-aggregation/), 2026-04-30, accessed 2026-07-23 (vendor/client-team blog reporting its own measurements): 4-second slot, 800 ms sub-slot, 750 ms soft deadline, mean per-proof aggregation time 2.3 seconds over a 400-slot run on a Skylake-SP server. The 3x came from compiler target flags (2.7x native compilation, 3.1x x86-64-v3), not algorithmic change - a direct warning that build flags can dominate a benchmark harness. Companion explainer [here](https://blog.lambdaclass.com/ethereum-signature-schemes-explained-ecdsa-bls-xmss-and-post-quantum-leansig-with-rust-code-examples/), 2026-02-09, accessed 2026-07-23: leanSig on M4 Max signing about 0.53 ms, verification about 0.19 ms, signature about 3 kB, aggregate proof about 100 KB.
- **Starknet account abstraction with Falcon.** s2morrow and the OpenZeppelin verifiers above are the Starknet-side artifacts; both report Cairo execution cost (steps, L2 gas), not Stwo prover cost.
- **Classical credential chains in a zkVM, as a same-generation control.** zk-X509, [arXiv:2603.25190](https://arxiv.org/abs/2603.25190), v2 2026-03-31, accessed 2026-07-23. SP1 v6.0.1 on an Apple M-series CPU: ECDSA P-256 11,803,639 cycles, RSA-2048 17,399,633 cycles, RSA-2048 + CRL 23,163,293 cycles, ECDSA P-384 47,775,211 cycles. On-chain Groth16 register about 300,000 gas. **These are execute-mode cycle counts with no proof generated**; the only proving figure in the paper is a single "~5 minutes" on an Apple M1 CPU, with a GPU figure explicitly labelled "estimated". The paper has zero post-quantum content and no batching.

Covers SHA256, Poseidon, Fibonacci; ECDSA/BLS listed as TBA. Example rows: SHA256 over 1 KB gives Jolt 62,231 cycles / 2.199 s / 401,116 B and SP1 71,249 cycles / 17.6 s / 265,691 B. **There is no post-quantum workload of any kind.**

An attribution of PQScale to the AURORA proving system came from a search extract, not the blog body, and is **unverified**.
- **[github.com/qryptalabs/qryptalabs](https://github.com/qryptalabs/qryptalabs)**, accessed 2026-07-23. Claims Dilithium/Falcon verification off-chain via SP1 but publishes zero cycle counts, prover times, proof sizes, or gas, and does not state which scheme is wired up. Cite only as evidence that nobody has published SP1 Falcon numbers.

## 3. The Starknet plus PQ gap, stated precisely

Stated as narrowly as the evidence supports:

1. **Falcon-512 and SPHINCS+-128s verifiers already exist in Cairo and are already measured in Cairo execution units, but nobody has published the Stwo prover cost.** s2morrow has both verifiers checked DONE and "Stwo proving benchmarks" explicitly UNCHECKED ([repo](https://github.com/starkware-bitcoin/s2morrow), accessed 2026-07-23). OpenZeppelin publishes L2 gas and Cairo steps for five Falcon-512 variants but no prover seconds, no prover RAM, and no proof size ([repo](https://github.com/OpenZeppelin/cairo-pq-verifiers), accessed 2026-07-23). The gap is on the **proving** side, not the verifier-implementation side.
2. **ML-DSA-44 in Cairo is not measured at all.** OpenZeppelin's ML-DSA-44 verifier is stubbed ([repo](https://github.com/OpenZeppelin/cairo-pq-verifiers), accessed 2026-07-23).
3. **No FIPS-conformant, full-path Falcon cost has been published in any proof system.** Every fast published number relaxes the standard somewhere: Miden substitutes RPO256 for SHAKE-256 and excludes decompression from the VM ([PR #369](https://github.com/0xPolygonMiden/miden-vm/pull/369), accessed 2026-07-23); OpenZeppelin's own table shows the SHAKE-256 variant at 312,892 steps against 97,681 for the Poseidon-native one, a 3.2x standards-compliance tax that no source states as a caveat.
4. **No same-harness classical control exists for PQ signatures on Starknet's prover.** The closest is s2morrow.xyz's L2-gas-only secp256r1/secp256k1/Garaga comparison ([page](https://www.s2morrow.xyz/), accessed 2026-07-23) and OpenZeppelin's ECDSA-STARK control, for which no number appeared in the retrieved table.
5. **No peak memory figure exists for Falcon or ML-DSA in any proof system.** The only PQ-in-STARK memory figures found anywhere are for hash-based schemes: ePrint 2021/1048's 0.9-9.5 GB range ([paper](https://eprint.iacr.org/2021/1048), accessed 2026-07-23) and Lemur's remark that a naively materialized HVC tree at tree height 20 "occupies many GB of RAM" ([paper](https://eprint.iacr.org/2026/1161), accessed 2026-07-23).
6. **Arithmetization coverage is thin.** No Halo2, Plonky2, Plonky3, gnark, Noir, or Circom Falcon verifier was found. Existing coverage is Arkworks R1CS, Jellyfish Plonk (unbenchmarked), Cairo/Stwo (unbenchmarked on the prover side), Miden MASM, and LaBRADOR's native lattice relation.

## 4. Honest positioning

**The broad novelty claim was refuted. Lead with that.**

The claim tested was: "No one has published clean, reproducible numbers for amortized STARK/zkVM prover cost per transaction when verifying post-quantum signatures in-circuit, as a function of batch size." Four independent adversarial reviews returned **refuted**, at medium confidence each. The strongest counterevidence:

- **ePrint 2021/1048** ([link](https://eprint.iacr.org/2021/1048), AsiaCCS 2022, accessed 2026-07-23) publishes exactly that artifact: prover wall-clock, prover RAM, and proof size at batch sizes 128 / 256 / 512 / 1024, at two security levels, on named hardware, for a post-quantum hash-based signature verified in-circuit in a STARK, with the AIR open-sourced in Winterfell and independently re-tabulated there. This is roughly five years old and peer-reviewed.
- **leanBench** ([link](https://github.com/leanEthereum/leanBench), accessed 2026-07-23) is a live, public, seed-pinned harness sweeping flat aggregation over 125-1000 signatures plus tree aggregation, recording timings, CPU and memory per hardware fingerprint, for a PQ hash-based signature inside a STARK zkVM.
- **HAPPIER** ([link](https://link.springer.com/chapter/10.1007/978-3-032-15541-2_1), accessed 2026-07-23) aggregates a PQ hash-based signature inside RISC Zero up to 2^16 signatures. **Its full text was not read (auth wall), so whether it tabulates prover seconds against batch size is unverified.** This is the single largest live refutation risk.

**The headline finding such a benchmark would produce is also already published, twice, and it is negative.** ePrint 2021/1048's Table 3 shows prover time and RAM growing linearly in batch size, so amortized prover cost per signature is essentially flat (derived: 19.5 / 19.9 / 20.5 / 19.2 ms per signature at n = 128/256/512/1024). ZK-ACE's Table 7 ([arXiv:2603.07974](https://arxiv.org/abs/2603.07974), accessed 2026-07-23) reproduces the same shape on a modern Circle STARK: 14.29 to 14.77 ms/tx across N = 1 to 16 while proof size falls from 122 KB to 13 KB per tx. **Only proof bytes amortize. Presenting "cost per transaction falls with batch size" as a discovery would be wrong and would be caught immediately.**

**What is genuinely new here** is the intersection, and only the intersection:

> Measured, same-harness prover cost - wall clock, peak RAM, and proof size on named hardware - for **FIPS-conformant, full-path** verification of the three NIST-standardized schemes (**ML-DSA-44** per FIPS 204, **FN-DSA/Falcon-512** per draft FIPS 206, **SLH-DSA-128s** per FIPS 205) inside a general-purpose zkVM or Cairo/Stwo, swept across batch size, with a **classical signature control in the same harness**.

Every qualifier is load-bearing. The existing batch-vs-prover-cost literature uses Lamport+, XMSS, leanSig, or purpose-built SNARK-friendly schemes, never the FIPS trio. The Falcon literature publishes sizes and constraint counts but not prover seconds. Miden and Poseidon-native Cairo variants relax the standard. Nobody has published Falcon against Ed25519 against ECDSA proved by the same prover on the same machine.

**What is reproduction, and must be labelled as such:**
- The methodology (AIR or zkVM program, sweep batch size, report prover seconds / RAM / proof bytes on named hardware) is ePrint 2021/1048's, from 2021.
- The flat-amortization result is a re-confirmation, not a discovery.
- Constraint-count reference points for Falcon are falcon-r1cs's, from 2022, unchanged since the repo was archived.
- Cairo step counts for Falcon-512 are OpenZeppelin's, from 2026.

**What others have already done and should not be re-claimed:** hash-based aggregation in a STARK (Khaburzaniya et al., 2021), hash-based aggregation in RISC Zero (HAPPIER, 2025), Falcon aggregation in a lattice proof system (LaBRADOR, 2024), the continuous cross-hardware PQ aggregation benchmark (leanBench, live), and the cross-zkVM evaluation of aggregation backends (leanroadmap's research track, roughly 50% complete).

**Three disclosures that must accompany any published result**, or the numbers will be attacked:
1. No mainstream zkVM ships a lattice/NTT precompile, so a PQ-vs-Ed25519 comparison in SP1 is unaccelerated against accelerated ([SP1 precompile docs](https://docs.succinct.xyz/docs/sp1/optimizing-programs/precompiles), accessed 2026-07-23).
2. Standards conformance costs about 3.2x in Cairo steps for Falcon-512 alone (SHAKE-256 312,892 vs Poseidon-native 97,681, [OpenZeppelin](https://github.com/OpenZeppelin/cairo-pq-verifiers), accessed 2026-07-23), so any comparison against a relaxed implementation must say so.
3. Build flags alone produced a 3x swing in a real aggregation deployment ([LambdaClass](https://blog.lambdaclass.com/ethlambda-how-we-got-a-3x-speedup-in-signature-aggregation/), 2026-04-30, accessed 2026-07-23), so compiler target must be pinned and reported.

## 5. Open questions this benchmark can actually answer

1. **What does FIPS conformance actually cost end to end?** OpenZeppelin measured the hashing substitution at 3.2x in Cairo steps for Falcon-512, but nobody has measured the full parse plus decompress plus hash-to-point plus verify path in prover seconds in any system.
2. **What is the real ML-DSA-44 in-zkVM cycle count?** The only figure in circulation is arithmetically self-inconsistent and comes from a crate containing no signature verifier. Even a single sound, reproducible cycle count would be new.
3. **What is the prover-side cost of the Falcon and SPHINCS+ Cairo verifiers that already exist?** s2morrow scoped this and left it unchecked; the verifiers are written and public.
4. **How do the three FIPS schemes rank against each other under one prover on one machine?** Falcon is smallest but hardest to arithmetize (NTT, Gaussian sampling, decompression); SLH-DSA is pure hashing and should be zkVM-friendly but has huge signatures; ML-DSA sits between. No published data ranks them under a single harness.
5. **What is peak prover RAM per scheme, and at what batch size does it exceed a commodity machine?** The only PQ-in-STARK RAM data anywhere is for hash-based schemes.
6. **Where is the crossover against the non-ZK alternatives?** ETHFALCON's 1.5M-3.9M gas Solidity verification and the EIP-7619 precompile proposal are the counterfactuals; LaBRADOR-native aggregation is the other. A same-machine cost comparison against both is missing.
7. **Does per-signature prover cost stay flat for lattice schemes as it does for hash-based ones?** The flat result is established for Lamport+ and for a small Poseidon2 circuit. Falcon's NTT structure and non-native arithmetic may behave differently, and confirming or refuting that is a legitimate, modest contribution.
8. **How much does the arithmetization choice matter?** Only R1CS, Cairo, MASM, and LaBRADOR-native coverage exists. A Plonky3 or Stwo number for the same scheme would be new.
